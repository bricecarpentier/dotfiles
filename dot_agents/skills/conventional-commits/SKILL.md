# Conventional Commits

Format: `type(scope): message`

## Types

| Type | When to use |
|------|-------------|
| `feat` | A new feature or user-facing behavior |
| `fix` | A bug fix |
| `docs` | Documentation-only changes |
| `style` | Formatting, whitespace, semicolons â€” no logic change |
| `refactor` | Code change that neither fixes a bug nor adds a feature |
| `perf` | Performance improvement |
| `test` | Adding or updating tests |
| `build` | Build system or external dependency changes |
| `ci` | CI configuration and scripts |
| `chore` | Maintenance tasks that don't fit other types |
| `revert` | Reverts a previous commit |

## Scope

- Scope is **optional** but encouraged â€” it identifies the area affected.
- Wrap in parentheses: `feat(auth): add SSO login`
- Use lowercase, short identifiers: app names, module names, layer names.
- Examples: `api`, `ui`, `db`, `cli`, `auth`, `billing`

## Message

- **Lowercase** first word, no period at the end.
- Use imperative mood: "add feature" not "added feature" or "adds feature".
- Keep the first line under 72 characters.

## Breaking Changes

Mark breaking changes with `!` after the scope (or type if no scope):

```
feat(api)!: remove deprecated /v1/users endpoint
```

Optionally include a `BREAKING CHANGE:` footer in the commit body for details.

## Formatting Rules for Code References

When the commit message mentions code identifiers â€” classes, functions, methods, constants, enums, variables, types, interfaces, file names, or CLI flags â€” wrap them in **backticks**:

```
fix(parser): handle `None` return from `get_token()`
refactor(db): rename `UserStatus` enum variants
feat(cli): add `--dry-run` flag to `deploy` command
chore: remove unused `MAX_RETRIES` constant
```

## Multi-line Commits

```
feat(auth): add OAuth2 PKCE flow

Implements the Authorization Code flow with PKCE for public clients.
Adds `PKCEChallenge` helper and updates `AuthProvider` config.

BREAKING CHANGE: `AuthProvider.init()` now requires a `redirectUri` parameter.
```

## Shell: Committing Messages with Backticks

**Never use `git commit -m` when the message contains backticks.** The shell
will mangle them (e.g. `\`` stored literally, or backtick contents interpreted
as command substitution).

Use a **quoted heredoc** with `git commit -F -` instead:

```bash
git commit -F - <<'EOF'
fix(parser): handle `None` return from `get_token()`

The `get_token()` helper was returning `None` on expiry but callers
assumed a non-null `Token`. Add an explicit check.

Generated by Mistral Vibe.
Co-Authored-By: Mistral Vibe <vibe@mistral.ai>
EOF
```

The single quotes around `'EOF'` prevent all shell expansion inside the heredoc.

## Examples

```
feat(billing): add invoice PDF export
fix(api): return 404 when `Workspace` not found
docs: update README setup instructions
style(ui): fix indentation in `Sidebar` component
refactor: extract `validateEmail()` into shared utils

test(auth): cover `refreshToken()` expiry edge case
build: upgrade `vite` to v6
ci: add deploy step for staging
chore: remove deprecated `legacyMode` flag
feat(api)!: require `organizationId` in all endpoints
```
